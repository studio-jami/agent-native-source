# Toolkit

`@agent-native/core` owns the foundational runtime: actions, app state, auth,
SQL, server plugins, agent chat, sharing access checks, and collaboration
transport. `@agent-native/toolkit` owns reusable app-building pieces that sit on
top of that runtime: UI primitives, app-local adapters, shell helpers, sharing
badges, and collaboration UI.

Apps should compose those pieces through local adapters. That keeps app UI easy
to customize without copying Toolkit implementation code.

## What Ships Today {#what-ships-today}

- **UI primitives**: shadcn-style components under
  `@agent-native/toolkit/ui/*`.
- **App adapters**: template-local wrappers such as
  `@/components/ui/button` that re-export Toolkit primitives until an app wants
  to customize them.
- **ToolkitProvider**: an override layer that lets reusable Toolkit UI use an
  app's local primitives, starting with `Button`.
- **Collaboration UI**: presence avatars, live cursors, remote selection rings,
  recent edit highlights, and agent presence chips under
  `@agent-native/toolkit/collab-ui`.
- **Sharing UI**: the reusable `VisibilityBadge` under
  `@agent-native/toolkit/sharing`.
- **Shell and hooks**: app-shell header actions plus shared hooks such as
  `useIsMobile` and `useToast`.

## Compatibility {#compatibility}

Existing imports from `@agent-native/core/client` keep working. Core now exposes
thin re-export shims for Toolkit-owned visual pieces while keeping runtime
behavior in core.

Those core shims are temporary migration support, not the long-term dependency
direction. The next major can remove them; after that, reusable app-building
packages should import Toolkit directly, and behaviorful Toolkit kits may compose
core runtime APIs as peer/runtime dependencies instead of pushing more UI back
into core.

Extract future behaviorful kits one at a time before broadening the split.
Sharing is the first candidate to validate that model because it exercises
access checks, action-backed data, and share-link UI together.

That means this keeps working:

```tsx
import { PresenceBar, VisibilityBadge } from "@agent-native/core/client";
```

New reusable app code should prefer Toolkit or app-local adapters:

```tsx
import { PresenceBar } from "@agent-native/toolkit/collab-ui";
import { Button } from "@/components/ui/button";
```

## Boundary {#boundary}

Move code into Toolkit when it is reusable app-building UI or a pure UI contract.
Keep code in core when it owns access control, SQL, actions, request context,
Yjs transport, auth, or other runtime guarantees.

Good Toolkit candidates:

- Components that render the same interaction across apps.
- Display-only helpers and structural prop types.
- Base components that receive data and callbacks from an app or core wrapper.
- App-shell hooks and adapters that do not own domain data.

Keep app-local:

- Domain-specific workflows.
- Resource-specific mutations and routes.
- Custom visual systems an app intentionally owns.

Keep core-owned:

- Sharing actions and access checks.
- Collaboration storage, awareness transport, and Yjs document sync.
- Auth, sessions, app state, SQL, and server plugins.
