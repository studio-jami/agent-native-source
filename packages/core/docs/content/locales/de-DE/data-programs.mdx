# Datenprogramme

Ein **Datenprogramm** ist ein benanntes, gespeichertes, vom Agenten verfasstes JS-Skript, das Daten abruft, verknüpft, filtert oder aggregiert – Provider-APIs über `providerFetch` / `providerFetchAll` / `providerSearchAll`, App-Daten über `appAction`, oder Resources-gestützte Workspace-Dateien – und genau einmal `emit(rows, schema)` mit dem Ergebnis aufruft. Das Ergebnis wird in SQL zwischengespeichert, und jede Chart-/Tabellenkomponente kann es rendern, aktualisiert nach einer von Ihnen gewählten Richtlinie statt bei jedem Seitenaufruf.

Es ist automatisch überall verfügbar, wo [`run-code`](/docs/actions) verfügbar ist – es gibt nichts, das Sie pro App einbinden oder konfigurieren müssen. Wenn Ihre App run-code bereits aktiviert hat, sind `save-data-program`, `preview-data-program`, `run-data-program`, `list-data-programs`, `get-data-program` und `delete-data-program` bereits registrierte actions.

## Warum es das gibt {#why}

Ohne ein gespeichertes Datenprogramm endet „baue eine Live-Ansicht von X" meist an einem von zwei schlechten Orten:

- **Eine fest codierte action pro Anbieter/Abfrageform.** Brüchig, und sie kann nicht jede benutzerdefinierte Eigenschaft jedes Kunden, einen anbieterübergreifenden Join oder einen willkürlichen Filter antizipieren, an den Sie nicht gedacht haben.
- **Dieselbe Abruf-/Join-Logik bei jedem Seitenaufruf von Grund auf neu ausführen.** Langsam, und es verbraucht bei jedem Blick auf ein Diagramm erneut dasselbe Provider-API-Kontingent.

Ein Datenprogramm ist der Mittelweg: Schreiben Sie die Abruf-/Join-/Aggregationslogik einmal als gewöhnlichen `run-code`, speichern Sie sie, und lassen Sie die Cache- und Aktualisierungsrichtlinie entscheiden, wie oft sie tatsächlich erneut ausgeführt werden muss.

## Wann Sie eines verwenden sollten {#when-to-use-one}

| Situation                                                                                                                                                                                                     | Greifen Sie zu                                                                                     |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| Ein Standard-Rollup hat bereits eine erstklassige action (ein vorgefertigter `hubspot-deals`-Filter, eine eingebaute Metrik)                                                                                  | Die vorhandene action — verpacken Sie sie nicht in ein Programm                                    |
| Eine einmalige Chat-Frage, die Sie nie wieder ansehen werden                                                                                                                                                  | Einfaches `run-code` oder `provider-api-request` — beantworten Sie es direkt, speichern Sie nichts |
| Eine Erweiterung benötigt ein kleines, interaktives Widget pro Benutzer mit eigener UI                                                                                                                        | [Erweiterungen](/docs/extensions) — ein Datenprogramm hat keine eigene UI                          |
| Eine Kohorte benötigt einen willkürlichen Eigenschaftsfilter, eine IN-Suche über benutzerdefinierte Werte oder einen Join über zwei Provider hinweg, **und** jemand möchte es mit aktuellen Daten wiedersehen | Ein Datenprogramm                                                                                  |
| Ein Dashboard-Panel soll dieses verknüpfte/aggregierte Ergebnis anzeigen und nach der Cache-Richtlinie aktualisieren                                                                                          | Ein Datenprogramm, gebunden an ein `"program"`-Panel                                               |

Ein Vergleich mit den anderen run-code-förmigen Optionen:

|                                                    | Einmaliges `run-code`     | Fest codierte Provider-action                                | Datenprogramm                                                    |
| -------------------------------------------------- | ------------------------- | ------------------------------------------------------------ | ---------------------------------------------------------------- |
| **Bleibt zwischen Aufrufen bestehen**              | Nein                      | N/A (es ist Quellcode)                                       | Ja — gespeicherte Zeile + Ergebnis-Cache                         |
| **Ergebnis wird zwischengespeichert/aktualisiert** | Nein, läuft jedes Mal neu | Kein eingebautes Caching                                     | Ja — `refreshMode`/`refreshTtlMs`                                |
| **Verarbeitet willkürliche Filter/Joins**          | Ja, aber danach verworfen | Nein — begrenzt auf das, was das Schema der action offenlegt | Ja — es ist immer noch gewöhnlicher Code                         |
| **An ein Dashboard-Panel bindbar**                 | Nein                      | Nur wenn eine Panel-Quelle dafür gebaut wurde                | Ja — die generische `"program"`-Quelle                           |
| **Erfordert ein Deployment**                       | Nein                      | Ja (neue action = Code-Änderung)                             | Nein — vom Agenten zur Laufzeit erstellt                         |
| **Zur Lesezeit verwendete Credentials**            | Die des Aufrufers         | Die des Aufrufers                                            | **Die des Betrachters**, pro Lesevorgang aufgelöst (siehe unten) |

## Autoren-Workflow {#authoring-workflow}

1. **Zuerst im Chat prototypisieren.** Schreiben Sie die Abruf-/Join-/Aggregationslogik als normales `run-code`-Skript und bestätigen Sie, dass es die erwarteten Zeilen zurückgibt.
2. **Ändern Sie es so, dass es genau einmal `emit(rows, schema?)` aufruft**, am Ende, statt das Ergebnis zurückzugeben oder auszugeben.
3. **Speichern Sie es** mit `save-data-program`. Dies **führt einen Dry-Run des Codes mit `defaultParams` durch, bevor es gespeichert wird**, und lehnt das Speichern mit einem strukturierten Fehler ab, falls es fehlschlägt, sodass ein defektes Programm nie gespeichert wird. Bei Erfolg gibt es `{ programId, rowCount, columns, sampleRows }` zurück — betrachten Sie das als Beweis, dass das Programm die erwarteten Zeilen liefert, bevor Sie dem Benutzer sagen, dass es bereit ist.
4. **Binden Sie ein Dashboard-Panel daran**, oder rufen Sie `run-data-program` direkt von Ihrem eigenen Lesepfad aus auf. Es gibt keine dashboard-spezifische Verkabelung, die in die Primitive selbst eingebaut ist — die `"program"`-Panel-Quelle der Analytics-Vorlage (unten) ist eine Anwendung davon, kein erforderliches Muster.
5. **Iterieren Sie, ohne zu speichern**, über `preview-data-program({ code, params? })` — derselbe Dry-Run-Pfad, keine gespeicherte Zeile.

### Ein realistisches Beispiel

Angenommen, eine App möchte eine Live-Tabelle „aktuelle hochwertige Support-Tickets" aus einem Support-Provider, gefiltert nach einem Schwellenwert, der von keiner eingebauten action offengelegt wird:

```ts
// The program's stored `code` — ordinary run-code, with one emit() call.
const minValue = Number(params.minValue || 5000);

const tickets = await providerFetchAll("acme-support", "/v1/tickets", {
  query: { status: "open" },
  itemsPath: "data",
  pagination: {
    cursorParam: "cursor",
    nextCursorPath: "next_cursor",
    maxPages: 10,
  },
});

const rows = (tickets.items || [])
  .filter((t) => Number(t.contract_value || 0) >= minValue)
  .map((t) => ({
    ticket_id: t.id,
    subject: t.subject,
    account: t.account_name,
    contract_value: Number(t.contract_value || 0),
    opened_at: t.created_at,
  }));

emit(rows, [
  { name: "ticket_id", type: "string" },
  { name: "subject", type: "string" },
  { name: "account", type: "string" },
  { name: "contract_value", type: "number" },
  { name: "opened_at", type: "string" },
]);
```

Beim Speichern:

```ts
await appAction("save-data-program", {
  name: "high-value-open-tickets",
  title: "High-Value Open Tickets",
  description:
    "Open support tickets above a configurable contract-value threshold.",
  code: "/* the script above */",
  defaultParams: { minValue: 5000 },
  refreshMode: "ttl",
  refreshTtlMs: 300_000,
});
```

Die Antwort enthält `{ programId, rowCount, columns, sampleRows }` — der Proof-of-Done des Dry-Runs. Von dort aus führt `run-data-program({ programId, params: { minValue: 10000 } })` es für einen anderen Schwellenwert erneut aus (oder liefert den Cache), und ein Dashboard-Panel kann sich auf dieselbe Weise an `programId` binden.

## Die Sandbox-Oberfläche {#sandbox-surface}

Ein Datenprogramm läuft durch genau dieselbe `executeSandboxCode`-Sandbox wie `run-code` — es gibt keinen neuen Sandboxing-, Credential-, SSRF- oder Kontingent-Code für diese Primitive. Es erhält dieselben Globals (`providerFetch`, `providerFetchAll`, `providerSearchAll`, `appAction`, `webFetch`, `workspace*`-Resources-Helfer) plus zwei Ergänzungen aus einer kleinen, vom Framework injizierten Präambel:

- **`params`** — ein **eingefrorenes** globales Objekt, das die Parameter für diesen Lauf enthält (`defaultParams`, zusammengeführt mit dem, was an `run-data-program` oder den Panel-Deskriptor übergeben wurde). Lesen ist in Ordnung; Mutieren wirft einen Fehler.
- **`emit(rows, schema?)`** — rufen Sie dies **genau einmal** auf, am Ende. Ein zweiter Aufruf wirft einen Fehler. `rows` muss ein Array von einfachen Objekten sein. `schema`, sofern angegeben, ist `{ name: string, type: string }[]`; wenn weggelassen, wird es aus den ersten 50 Zeilen abgeleitet (eine Spalte ist `"json"`, wenn Zeilen im Sample beim primitiven Typ nicht übereinstimmen, andernfalls `"number"` / `"string"` / `"boolean"`). `console.log` bleibt vollständig frei für Debugging — der Runner erfasst kombiniertes stdout/stderr und entfernt beim Parsen des Ergebnisses nur die Sentinel-Zeile, die `emit()` schreibt.

Bei jedem Lauf durchgesetzte Obergrenzen (nicht pro Programm konfigurierbar):

| Grenze                                                    | Wert                                                                                                                                                                                         |
| --------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Max. emittierte Zeilen                                    | 10.000 — Zeilen darüber hinaus werden verworfen, `truncated: true`                                                                                                                           |
| Max. emittierte Ergebnisgröße                             | 4 MiB — Zeilen werden vom Ende entfernt, um zu passen; eine einzelne Zeile über der Obergrenze ist ein harter `result_too_large`-Fehler, da es nichts gibt, das sicher gekürzt werden könnte |
| Max. aktive (nicht archivierte) Programme pro App         | 200 — archivieren Sie ungenutzte, um Platz zu schaffen                                                                                                                                       |
| Minimales Aktualisierungs-TTL                             | 60.000 ms                                                                                                                                                                                    |
| Aufbewahrte Laufzeilen pro (Programm, Params, Betrachter) | 5 neueste — ältere werden bei jedem Schreibvorgang automatisch bereinigt                                                                                                                     |

Kürzung ist immer explizit (`truncated: true` im Ergebnis) — nie ein stiller Verlust.

## Params und Dashboard-Panel-Bindung {#params-and-panel-binding}

Programme deklarieren ein optionales `paramsSchema` (JSON Schema, das akzeptierte Params beschreibt) und `defaultParams`. Ein Dashboard übernimmt die `"program"`-Panel-Quelle, indem es einen kleinen JSON-Deskriptor im Abfragefeld des Panels statt einer Abfragezeichenfolge platziert:

```json
{ "programId": "dp_a1b2c3d4e5f6", "params": { "minValue": 10000 } }
```

Das Panel rendert durch genau dieselben Chart-/Tabellenkomponenten wie jede andere Panel-Quelle — es erhält immer nur `{ rows, schema }` zurück. Ein Panel an ein Programm zu binden bedeutet nur „welches gespeicherte Programm, mit welchen Param-Überschreibungen"; es gibt keinen anderswo duplizierten dashboard-spezifischen Ausführungspfad.

## Aktualisierung, TTL und Caching-Semantik {#refresh-and-caching}

Jedes `(programId, paramsHash)`-Paar hat seine eigene Laufhistorie in `data_program_runs`, und `paramsHash` bezieht **den Betrachter** ein, nicht nur die Params (siehe den Sicherheitsabschnitt für den Grund). Was bei jedem Lesevorgang passiert:

| Situation                                                                                      | Verhalten                                                                                                                                                                                                                                |
| ---------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Frischer Cache (jünger als `refreshTtlMs`, oder `refreshMode: "manual"` mit vorherigem Erfolg) | Gibt sofort zwischengespeicherte Zeilen zurück, `cacheHit: true`                                                                                                                                                                         |
| Cache ist älter als das TTL                                                                    | Führt synchron erneut aus — ein 25s-Budget für Panel-View-Lesevorgänge, 120s für Agent-/manuelle Aufrufe — und ersetzt den Cache bei Erfolg                                                                                              |
| `refreshMode: "manual"`                                                                        | Aktualisiert nie automatisch; nur ein explizites `forceRefresh: true` (oder eine manuelle Aktualisierungs-UI-action) führt es erneut aus                                                                                                 |
| Noch kein Cache                                                                                | Führt synchron aus wie ein normaler erster Abruf                                                                                                                                                                                         |
| `background: true`-Programm, kein frischer Cache                                               | Liefert sofort den letzten guten Lauf mit `stale: true` und stellt eine dauerhafte Hintergrundausführung in die Warteschlange (10-Minuten-Budget); ein späterer Aufruf finalisiert das Ergebnis, sobald die Ausführung abgeschlossen ist |
| Hintergrundprogramm läuft noch, kein vorheriger Erfolg                                         | `background_pending`-Fehler — zeigt einen expliziten Zustand „läuft noch" an, nie ein leeres Ergebnis                                                                                                                                    |
| Ausführung schlägt fehl (Timeout, Sandbox-Fehler, falsche `emit()`-Form)                       | Ein strukturierter `{ code, message }`-Fehler; wenn ein vorheriger Erfolg existiert, wird er als `lastGoodRun` angehängt, sodass ein Panel veraltete Daten anzeigen kann, statt leer zu bleiben                                          |
| Programm wurde archiviert (soft-deleted über `delete-data-program`)                            | Expliziter `archived`-Fehler — nie ein stilles leeres Panel                                                                                                                                                                              |

Greifen Sie nur für Programme zu `background: true`, die routinemäßig das Vordergrund-Timeout überschreiten — große Multi-Provider-Joins, tiefe Paginierung. Alles andere sollte im Vordergrund bleiben: Es ist einfacher zu verstehen und liefert schneller frische Daten.

### Warum der Cache pro Betrachter gescopet ist {#per-viewer-cache}

`providerFetch` innerhalb eines Programms löst die Authentifizierung anhand des **eigenen Anfragekontexts des aufrufenden Betrachters** auf — nie des Programmautors. Das gilt unabhängig davon, ob der Aufruf vom Agenten, einem `run-data-program`-Aufruf oder einem von einem Teammitglied angesehenen Dashboard-Panel kommt. Zwei verschiedene Betrachter, die dasselbe Programm mit denselben Params ausführen, können legitim zwei unterschiedliche Ergebnisse erhalten: Einer hat einen konfigurierten HubSpot-Schlüssel und der andere nicht, oder die Provider-Berechtigungen auf Zeilenebene unterscheiden sich pro Person.

Deshalb bezieht `hashDataProgramParams` die Identität des Aufrufers zusätzlich zu den kanonisierten Params in den Cache-Schlüssel ein. Das ist am wichtigsten für **geteilte Dashboards**: Wäre der Cache nur nach Params geschlüsselt, würde die erste Person, die ein Panel ansieht, still bestimmen, was jeder andere Betrachter sieht — einschließlich eines Betrachters ohne die Credentials, die dieses zwischengespeicherte Ergebnis erzeugt haben. Die Scopierung pro Betrachter bedeutet, dass ein Teammitglied ohne Credential seinen eigenen Auth-Fehler auf diesem Panel sieht, nicht die Daten einer anderen Person.

## Sicherheitsmodell {#security}

<Callout id="doc-block-dp-security" tone="success">

**Rohe Provider-Tokens erreichen nie das Modell oder den Browser.** Das Einzige, was die Sandbox jemals verlässt, ist das `{ rows, schema }`, das ein `emit()`-Aufruf erzeugt — dieselbe Vertrauensgrenze wie bei jeder anderen run-code-Ausführung und jeder anderen Dashboard-Panel-Quelle.

</Callout>

- **Nur serverseitig.** Der Code eines Datenprogramms wird ausschließlich innerhalb von `executeSandboxCode` auf dem Server ausgeführt. Weder der gespeicherte Quellcode noch irgendein Credential, das er berührt, wird an den Browser oder das Modell gesendet — nur die emittierten Zeilen/Schema und kompakte Laufmetadaten (Zeilenanzahl, Spalten, Beispielzeilen) werden es.
- **Credentials sind immer die des aufrufenden Betrachters, nie die des ursprünglichen Autors des Programms.** Siehe [Aktualisierung, TTL und Caching-Semantik](#per-viewer-cache) oben, warum dies auch den Cache-Schlüssel bestimmt.
- **`allowPublic: false` ist eine feste Invariante, kein Standardwert.** Datenprogramme registrieren sich bei der Sharing-Registry auf dieselbe Weise wie [Erweiterungen](/docs/extensions#security), mit deaktivierter öffentlicher Freigabe. Ein öffentliches Programm würde jedem authentifizierten Benutzer erlauben, beliebigen gespeicherten Code unter seinem eigenen Token auszuführen — das Teilen eines Programms ist auf Ihre Organisation beschränkt und bedeutet immer nur „Teammitglieder können das zwischengespeicherte Ergebnis ansehen und erneute Läufe auslösen", nie „das allgemeine Internet kann dies ausführen."
- **Mutierende und ausführende actions sind nicht tool-aufrufbar von der sandboxed Extension-/Iframe-Bridge aus.** `save-data-program`, `preview-data-program`, `run-data-program` und `delete-data-program` sind `toolCallable: false` — nur der Agent oder ein serverseitiger Lesepfad kann sie aufrufen, nie ein Extension-Iframe unter der Sitzung des Öffners. `list-data-programs` und `get-data-program` sind schreibgeschützt und über dieselben Sharing-Helfer wie jede andere ownable Ressource gescopet.
- **Programme sind auf die App beschränkt, die sie erstellt hat.** Die Übergabe von `appId` an eine Suche bedeutet, dass ein in einer App erstelltes Programm in einer Shared-Database-Bereitstellung nicht vom Agent-Chat einer anderen App aus ausführbar ist, selbst für einen Aufrufer, der sonst Zugriff auf Zeilenebene (Owner/Org/Share) darauf hätte.
- **Binden Sie sich nur an ein Programm oder sehen Sie es nur an, dem Sie vertrauen.** Da die Ausführung immer die eigenen Credentials des Betrachters verwendet, ist das Ansehen des Ergebnisses eines geteilten Programms genau wie das Öffnen der Erweiterung einer anderen Person — vertrauen Sie dem Code, nicht nur der Zugriffsberechtigung.

**`risk-meeting-cohort.js`** — sucht nach einer konfigurierbaren Kohorte von CRM-Deals anhand einer willkürlichen `risk_status`-Eigenschaft (eine IN-Suche über benutzerdefinierte Werte, die eine vorgefertigte action nicht antizipieren kann), löst das Unternehmen jedes Deals über eine gebündelte Zuordnungssuche auf, löst die Domain jedes Unternehmens auf, ruft die Kontostimmung eines zweiten, nicht verwandten Providers ab und verknüpft die beiden nach Domain:

```js
// params: { riskStatuses?: string[], enterpriseOnly?: boolean }
const riskStatuses = params.riskStatuses?.length
  ? params.riskStatuses
  : ["On the Radar", "Churn Risk", "Confirmed Churn", "No Save Attempted"];

// 1. Arbitrary property IN-search — not expressible through a canned filter.
const dealSearch = await providerFetchAll("crm-provider", "/deals/search", {
  method: "POST",
  body: {
    filterGroups: [
      {
        filters: [
          { propertyName: "risk_status", operator: "IN", values: riskStatuses },
        ],
      },
    ],
  },
  itemsPath: "results",
  pagination: {
    cursorBodyPath: "after",
    nextCursorPath: "paging.next.after",
    maxPages: 20,
  },
});

// 2-3. Batched deal -> company -> domain resolution (elided here; see the
// real seed file for the full batching helper).

// 4. A second, unrelated provider — zero bespoke glue action required.
const accounts = await providerFetchAll("support-provider", "/accounts", {
  itemsPath: "data",
  pagination: {
    cursorParam: "cursor",
    nextCursorPath: "pagination.cursor",
    maxPages: 20,
  },
});

// 5. Join + emit.
const rows = dealSearch.items.map((deal) => ({
  deal_id: deal.id,
  risk_status: deal.properties.risk_status,
  arr: Number(deal.properties.total_contract_value || 0),
  // ...domain lookup + sentiment join...
}));

emit(rows, [
  { name: "deal_id", type: "string" },
  { name: "risk_status", type: "string" },
  { name: "arr", type: "number" },
]);
```

**`risk-meeting-pylon-early-warning.js`** — die ergänzende Ansicht: Konten, die bereits durch Support-Stimmung als gefährdet markiert wurden und **noch nicht** in der obigen CRM-Kohorte aufgetaucht sind. Diese Lücke — Support-Signal, das dem CRM-Signal voraus ist — erfordert das Ausschließen der Kohorte eines Providers von der eines anderen, genau die Art von anbieterübergreifender Mengenlogik, die keine einzelne Anbieter-action ausdrücken kann.

Beide Programme werden von `ensure-risk-meeting-dashboard` installiert und aktuell gehalten, welches:

- Jedes Programm nach seinem stabilen `name` aktualisiert, sodass ein erneutes Ausführen des Installers dieselben Zeilen aktualisiert, statt sie zu duplizieren.
- Keine CRM-/Support-Credentials zur Installationszeit benötigt — die Provider-Authentifizierung wird pro Betrachter erst aufgelöst, wenn ein Panel tatsächlich gerendert oder `run-data-program` aufgerufen wird, konsistent mit der obigen Cache-Scopierung pro Betrachter.
- Ein Zwei-Panel-Dashboard erstellt, dessen Panels `source: "program"` verwenden, gebunden an die id jedes Programms.

Siehe die `data-programs`-Skill für die vollständig kommentierte Quelle.

## Wie andere Apps die Panel-Quelle übernehmen {#adopting-the-panel-source}

Die `"program"`-Dashboard-Panel-Quelle ist eine Anwendung der Primitive, vollständig implementiert in der eigenen Panel-Abfrageschicht der Host-App — nichts Dashboard-spezifisches ist in `@agent-native/core/data-programs` selbst eingebaut. Die Implementierung der Analytics-Vorlage ist eine nützliche Referenz für die eigene Verkabelung:

- Die `sql`-Spalte des Panels enthält einen kleinen JSON-Deskriptor — `{ programId, params? }` — statt einer Abfragezeichenfolge.
- Das Rendern eines Panels ruft `runDataProgram` mit `triggeredBy: "panel_view"` und der Identität des aktuellen Betrachters auf, sodass der Cache pro Betrachter und die Credential-Auflösung automatisch gelten.
- Bei Erfolg erhält das Panel `{ rows, schema }` — dieselbe Form, die jede andere Panel-Quelle zurückgibt, sodass es unverändert durch die vorhandenen Chart-/Tabellenkomponenten fließt.
- Bei einem Fehler wird ein `lastGoodRun` einem defekten Panel vorgezogen, wann immer eines existiert; nur wenn kein vorheriger Erfolg existiert, zeigt das Panel einen strukturierten Fehler an (mit freundlicherer Formulierung für gängige Codes wie `access_denied`, `archived` und `background_pending`).

Jede App kann dieselbe Anwendung für ihre eigenen Dashboards/Tabellen implementieren: Rufen Sie `run-data-program` auf (oder `runDataProgram` direkt, wenn Sie sich im selben Serverprozess befinden) von Ihrem eigenen Lesepfad aus, und übergeben Sie die zurückgegebenen `rows`/`schema` an Ihre eigenen Rendering-Komponenten.
