# برامج البيانات

**برنامج البيانات (data program)** هو نص JS مسمّى ومخزَّن يكتبه الـ agent، يقوم بجلب البيانات أو ربطها أو تصفيتها أو تجميعها — عبر واجهات provider APIs باستخدام `providerFetch` / `providerFetchAll` / `providerSearchAll`، أو بيانات الـ app عبر `appAction`، أو ملفات workspace المستندة إلى Resources — ويستدعي `emit(rows, schema)` مرة واحدة بالضبط مع النتيجة. تُخزَّن النتيجة مؤقتًا في SQL، ويمكن لأي مكوّن رسم بياني/جدول عرضها، مع التحديث وفق سياسة تختارها بدلاً من إعادة الجلب في كل زيارة للصفحة.

يتوفر تلقائيًا أينما كان [`run-code`](/docs/actions) متاحًا — لا حاجة لتثبيته أو إعداده لكل app. إذا كان run-code مفعّلًا بالفعل في تطبيقك، فإن `save-data-program`، و`preview-data-program`، و`run-data-program`، و`list-data-programs`، و`get-data-program`، و`delete-data-program` هي بالفعل actions مسجَّلة.

## لماذا يوجد هذا {#why}

بدون برنامج بيانات مخزَّن، عادةً ما ينتهي "بناء عرض حي لـ X" في أحد مكانين سيئين:

- **action مبرمَجة مسبقًا لكل نوع مزود/شكل استعلام.** هشّة، ولا يمكنها توقّع الخصائص المخصصة لكل عميل، أو ربط عبر مزودين مختلفين، أو فلتر عشوائي لم تفكر في دعمه.
- **إعادة تشغيل منطق الجلب/الربط ذاته من الصفر في كل زيارة للصفحة.** بطيء، ويستهلك من جديد نفس حصة API الخاصة بالمزود كلما نظر أحدهم إلى رسم بياني.

برنامج البيانات هو الحل الوسط: اكتب منطق الجلب/الربط/التجميع مرة واحدة كـ `run-code` عادي، احفظه، ودع سياسة التخزين المؤقت والتحديث تقرر عدد المرات التي يحتاج فيها فعليًا لإعادة التشغيل.

## متى تستخدم واحدًا {#when-to-use-one}

| الحالة                                                                                                                                  | الحل الأنسب                                                                         |
| --------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| يوجد بالفعل action من الدرجة الأولى لتجميع قياسي (فلتر `hubspot-deals` جاهز، مقياس مدمج)                                                | الـ action الموجودة — لا تُغلّفها ببرنامج                                           |
| سؤال دردشة لمرة واحدة لن تنظر إليه مجددًا                                                                                               | استخدم `run-code` بسيط أو `provider-api-request` — أجب مباشرة، ولا تُخزّن شيئًا     |
| يحتاج امتداد (extension) إلى أداة تفاعلية صغيرة لكل مستخدم بواجهتها الخاصة                                                              | [الامتدادات (Extensions)](/docs/extensions) — لا تملك برامج البيانات واجهة خاصة بها |
| تحتاج مجموعة بيانات إلى فلتر خاصية عشوائي، أو بحث IN على قيم مخصصة، أو ربط بين مزودَين، **و**سيرغب أحدهم في رؤيتها مجددًا ببيانات حديثة | برنامج بيانات                                                                       |
| يجب أن تعرض لوحة dashboard تلك النتيجة المرتبطة/المجمّعة وتتحدث وفق سياسة التخزين المؤقت                                                | برنامج بيانات مرتبط بلوحة `"program"`                                               |

مقارنة مع الخيارات الأخرى على شاكلة run-code:

|                                            | `run-code` لمرة واحدة      | action مبرمَجة مسبقًا لمزود                   | برنامج بيانات                                           |
| ------------------------------------------ | -------------------------- | --------------------------------------------- | ------------------------------------------------------- |
| **يستمر بين الاستدعاءات**                  | لا                         | غير قابل للتطبيق (هو نفسه الكود المصدري)      | نعم — سطر مخزَّن + تخزين مؤقت للنتيجة                   |
| **النتيجة تُخزَّن مؤقتًا/تُحدَّث**         | لا، يُعاد تشغيله في كل مرة | لا يوجد تخزين مؤقت مدمج                       | نعم — `refreshMode`/`refreshTtlMs`                      |
| **يتعامل مع فلاتر/روابط عشوائية**          | نعم، لكن تُهمَل بعد ذلك    | لا — محدود بما يكشفه schema الخاص بالـ action | نعم — لا يزال كودًا عاديًا                              |
| **قابل للربط بلوحة dashboard**             | لا                         | فقط إذا بُني مصدر لوحة خصيصًا لذلك            | نعم — المصدر العام `"program"`                          |
| **يتطلب نشرًا (deploy)**                   | لا                         | نعم (action جديدة = تغيير في الكود)           | لا — يُنشئه الـ agent وقت التشغيل                       |
| **بيانات الاعتماد المستخدَمة عند القراءة** | بيانات المستدعي            | بيانات المستدعي                               | **بيانات المُشاهِد**، تُحل لكل عملية قراءة (انظر أدناه) |

## سير عمل الكتابة {#authoring-workflow}

1. **اصنع نموذجًا أوليًا في الدردشة أولاً.** اكتب منطق الجلب/الربط/التجميع كنص `run-code` عادي وتأكد من أنه يعيد الصفوف المتوقعة.
2. **عدّله ليستدعي `emit(rows, schema?)` مرة واحدة بالضبط** في النهاية، بدلاً من إعادة النتيجة أو طباعتها.
3. **احفظه** باستخدام `save-data-program`. هذا **يُجري تشغيلًا تجريبيًا (dry-run) للكود باستخدام `defaultParams` قبل الحفظ** ويرفض الحفظ بخطأ منظّم إذا فشل، بحيث لا يُخزَّن أبدًا برنامج معطوب. عند النجاح، تُعاد `{ programId, rowCount, columns, sampleRows }` — اعتبر ذلك دليلًا على أن البرنامج ينتج الصفوف المتوقعة قبل إخبار المستخدم بأنه جاهز.
4. **اربط لوحة dashboard به**، أو استدعِ `run-data-program` مباشرة من مسار القراءة الخاص بك. لا يوجد أي ربط خاص بلوحات المعلومات مدمج في هذه البنية الأساسية نفسها — مصدر لوحة `"program"` الخاص بقالب analytics (أدناه) هو أحد أشكال التبني، وليس نمطًا إلزاميًا.
5. **كرّر التجربة دون حفظ** عبر `preview-data-program({ code, params? })` — نفس مسار التشغيل التجريبي، دون سطر مخزَّن.

### مثال واقعي

لنفترض أن تطبيقًا يريد جدولًا حيًّا لـ"تذاكر الدعم عالية القيمة الأخيرة" مصدره مزود دعم، ومُصفًّى وفق عتبة لا تكشفها أي action مدمجة:

```ts
// The program's stored `code` — ordinary run-code, with one emit() call.
const minValue = Number(params.minValue || 5000);

const tickets = await providerFetchAll("acme-support", "/v1/tickets", {
  query: { status: "open" },
  itemsPath: "data",
  pagination: {
    cursorParam: "cursor",
    nextCursorPath: "next_cursor",
    maxPages: 10,
  },
});

const rows = (tickets.items || [])
  .filter((t) => Number(t.contract_value || 0) >= minValue)
  .map((t) => ({
    ticket_id: t.id,
    subject: t.subject,
    account: t.account_name,
    contract_value: Number(t.contract_value || 0),
    opened_at: t.created_at,
  }));

emit(rows, [
  { name: "ticket_id", type: "string" },
  { name: "subject", type: "string" },
  { name: "account", type: "string" },
  { name: "contract_value", type: "number" },
  { name: "opened_at", type: "string" },
]);
```

حفظه:

```ts
await appAction("save-data-program", {
  name: "high-value-open-tickets",
  title: "High-Value Open Tickets",
  description:
    "Open support tickets above a configurable contract-value threshold.",
  code: "/* the script above */",
  defaultParams: { minValue: 5000 },
  refreshMode: "ttl",
  refreshTtlMs: 300_000,
});
```

تحمل الاستجابة `{ programId, rowCount, columns, sampleRows }` — إثبات إتمام التشغيل التجريبي. من هناك، يعيد `run-data-program({ programId, params: { minValue: 10000 } })` تشغيله بعتبة مختلفة (أو يقدّم التخزين المؤقت)، ويمكن لأي لوحة dashboard الارتباط بـ `programId` بنفس الطريقة.

## سطح الصندوق الرملي (sandbox) {#sandbox-surface}

يعمل برنامج البيانات عبر نفس صندوق `executeSandboxCode` الرملي تمامًا الذي يستخدمه `run-code` — لا يوجد أي كود جديد خاص بالصندوق الرملي أو بيانات الاعتماد أو SSRF أو الحصص لهذه البنية الأساسية. يحصل على نفس المتغيرات العامة (`providerFetch`، `providerFetchAll`، `providerSearchAll`، `appAction`، `webFetch`، مساعدات Resources من نوع `workspace*`) بالإضافة إلى إضافتين من مقدمة صغيرة يحقنها الإطار (framework):

- **`params`** — كائن عام **مجمَّد (frozen)** يحمل معاملات هذا التشغيل (`defaultParams` مدمجة مع ما تم تمريره إلى `run-data-program` أو واصف اللوحة). قراءته آمنة؛ تعديله يطلق خطأ.
- **`emit(rows, schema?)`** — استدعِ هذا **مرة واحدة بالضبط**، في النهاية. الاستدعاء الثاني يطلق خطأ. يجب أن تكون `rows` مصفوفة من كائنات عادية (plain objects). `schema`، إن وُفّرت، تكون `{ name: string, type: string }[]`؛ وإن حُذفت يُستنتج من أول 50 صفًا (يكون العمود `"json"` إذا اختلفت الصفوف في النوع الأساسي ضمن العيّنة، وإلا فهو `"number"` / `"string"` / `"boolean"`). يبقى `console.log` حرًّا تمامًا لأغراض التصحيح — يلتقط المُشغِّل مخرجات stdout/stderr مجتمعة ولا يزيل سوى السطر الدال (sentinel) الذي يكتبه `emit()` عند تحليل النتيجة.

الحدود المفروضة على كل تشغيل (غير قابلة للتهيئة لكل برنامج على حدة):

| الحد                                                     | القيمة                                                                                                                                               |
| -------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| أقصى عدد صفوف يمكن إرسالها                               | 10,000 — تُحذف الصفوف الزائدة، `truncated: true`                                                                                                     |
| أقصى حجم للنتيجة المُرسَلة                               | 4 ميبيبايت — تُحذف الصفوف من النهاية لتلائم الحد؛ صفّ واحد يتجاوز الحد يُعد فشلًا صارمًا من نوع `result_too_large`، إذ لا يوجد ما يمكن اقتطاعه بأمان |
| أقصى عدد برامج نشطة (غير مؤرشفة) لكل تطبيق               | 200 — أرشِف غير المستخدَمة لتحرير مساحة                                                                                                              |
| الحد الأدنى لمدة التحديث (TTL)                           | 60,000 مللي ثانية                                                                                                                                    |
| عدد صفوف التشغيل المحفوظة لكل (برنامج، معاملات، مُشاهِد) | أحدث 5 — تُنظَّف الأقدم تلقائيًا مع كل عملية كتابة                                                                                                   |

الاقتطاع دائمًا صريح (`truncated: true` في النتيجة) — لا يحدث إسقاط صامت أبدًا.

## المعاملات وربط لوحة dashboard {#params-and-panel-binding}

يمكن للبرامج الإعلان عن `paramsSchema` اختياري (JSON Schema يصف المعاملات المقبولة) و`defaultParams`. تتبنى لوحة dashboard مصدر `"program"` بوضع واصف JSON صغير في حقل الاستعلام الخاص باللوحة بدلاً من سلسلة استعلام:

```json
{ "programId": "dp_a1b2c3d4e5f6", "params": { "minValue": 10000 } }
```

تُعرض اللوحة عبر نفس مكونات الرسم البياني/الجدول تمامًا التي يستخدمها أي مصدر لوحة آخر — لا تتلقى سوى `{ rows, schema }` دائمًا. ربط لوحة ببرنامج هو ببساطة تحديد "أي برنامج مخزَّن، وبأي تجاوزات للمعاملات"؛ لا يوجد أي مسار تنفيذ خاص بلوحات المعلومات مكرَّر في مكان آخر.

## دلالات التحديث ومدة الصلاحية (TTL) والتخزين المؤقت {#refresh-and-caching}

لكل زوج `(programId, paramsHash)` سجل تشغيل خاص به في `data_program_runs`، ويضم `paramsHash` **المُشاهِد** أيضًا وليس المعاملات فقط (انظر قسم الأمان لمعرفة السبب). ما يحدث في كل قراءة:

| الحالة                                                                            | السلوك                                                                                                                                                           |
| --------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| تخزين مؤقت حديث (أحدث من `refreshTtlMs`، أو `refreshMode: "manual"` مع نجاح سابق) | يعيد الصفوف المخزَّنة مؤقتًا فورًا، `cacheHit: true`                                                                                                             |
| التخزين المؤقت أقدم من مدة الصلاحية                                               | يُعاد تنفيذه بشكل متزامن — بميزانية 25 ثانية للقراءات المُطلَقة من عرض اللوحة، و120 ثانية لاستدعاءات الـ agent/اليدوية — ويستبدل التخزين المؤقت عند النجاح       |
| `refreshMode: "manual"`                                                           | لا يتحدّث تلقائيًا أبدًا؛ فقط `forceRefresh: true` صريح (أو action تحديث يدوي من الواجهة) يعيد تشغيله                                                            |
| لا يوجد تخزين مؤقت بعد                                                            | يُنفَّذ بشكل متزامن مثل أول جلب عادي                                                                                                                             |
| برنامج `background: true`، دون تخزين مؤقت حديث                                    | يقدّم آخر تشغيل ناجح فورًا مع `stale: true` ويُدرج تنفيذًا خلفيًا معمِّرًا في قائمة الانتظار (ميزانية 10 دقائق)؛ استدعاء لاحق يُنهي النتيجة بمجرد اكتمال التنفيذ |
| برنامج خلفي لا يزال قيد التشغيل، ودون نجاح سابق                                   | فشل `background_pending` — اعرض حالة صريحة "لا تزال قيد الحساب"، وليس نتيجة فارغة أبدًا                                                                          |
| فشل التنفيذ (انتهاء مهلة، خطأ في الصندوق الرملي، شكل `emit()` غير صالح)           | فشل منظّم `{ code, message }`؛ إذا وُجد نجاح سابق، يُرفَق كـ `lastGoodRun` بحيث يمكن للوحة عرض بيانات قديمة بدلاً من أن تبقى فارغة                               |
| البرنامج مؤرشَف (حذف ناعم عبر `delete-data-program`)                              | فشل صريح `archived` — لا تظهر أبدًا لوحة فارغة صامتة                                                                                                             |

استخدم `background: true` فقط للبرامج التي تتجاوز باستمرار مهلة الواجهة الأمامية — روابط ضخمة متعددة المزودين، ترقيم صفحات عميق. يجب أن يبقى كل شيء آخر في الواجهة الأمامية: أبسط في التفكير ويقدّم بيانات حديثة أسرع.

### لماذا يُحدَّد نطاق التخزين المؤقت وفق المُشاهِد {#per-viewer-cache}

يحل `providerFetch` داخل أي برنامج المصادقة باستخدام **سياق الطلب الخاص بالمُشاهِد المستدعي نفسه** — وليس سياق مؤلف البرنامج أبدًا. هذا صحيح سواء جاء الاستدعاء من الـ agent، أو من استدعاء `run-data-program`، أو من لوحة dashboard يشاهدها زميل في الفريق. يمكن لمشاهدَين مختلفين يشغّلان نفس البرنامج بنفس المعاملات الحصول شرعيًا على نتيجتين مختلفتين: أحدهما لديه مفتاح HubSpot مُعدّ والآخر ليس لديه، أو تختلف أذونات المزود على مستوى الصف حسب الشخص.

لهذا السبب، يضم `hashDataProgramParams` هوية المستدعي إلى مفتاح التخزين المؤقت إلى جانب المعاملات المُوحَّدة الشكل. هذا الأمر مهم بشكل خاص في **لوحات المعلومات المشتركة**: لو كان التخزين المؤقت مفهرَسًا بالمعاملات فقط، لكان أول شخص يشاهد لوحة يحدد بصمت ما سيراه كل مشاهد آخر، بمن فيهم مشاهد لا يملك بيانات الاعتماد التي أنتجت تلك النتيجة المخزَّنة مؤقتًا. تحديد النطاق حسب المُشاهِد يعني أن زميلًا في الفريق يفتقر إلى بيانات اعتماد سيرى خطأ المصادقة الخاص به على تلك اللوحة، وليس بيانات شخص آخر.

## نموذج الأمان {#security}

<Callout id="doc-block-dp-security" tone="success">

**لا تصل رموز (tokens) المزوّد الخام إلى النموذج أو المتصفح أبدًا.** الشيء الوحيد الذي يغادر الصندوق الرملي هو `{ rows, schema }` الذي ينتجه استدعاء `emit()` — نفس حدود الثقة المطبَّقة على أي تنفيذ run-code آخر وأي مصدر لوحة dashboard آخر.

</Callout>

- **من جانب الخادم فقط.** يُنفَّذ كود برنامج البيانات حصريًا داخل `executeSandboxCode` على الخادم. لا يُرسَل إلى المتصفح أو النموذج لا الكود المصدري المخزَّن ولا أي بيانات اعتماد يلمسها — يُرسَل فقط الصفوف/schema المُرسَلة وبيانات وصفية مضغوطة عن التشغيل (عدد الصفوف، الأعمدة، صفوف عيّنة).
- **بيانات الاعتماد هي دائمًا بيانات المُشاهِد المستدعي، وليست أبدًا بيانات المؤلف الأصلي للبرنامج.** انظر [دلالات التحديث ومدة الصلاحية والتخزين المؤقت](#per-viewer-cache) أعلاه لمعرفة سبب تحديد ذلك أيضًا لمفتاح التخزين المؤقت.
- **`allowPublic: false` هو ثابت صارم، وليس قيمة افتراضية.** تُسجَّل برامج البيانات في سجل المشاركة بنفس طريقة [الامتدادات](/docs/extensions#security)، مع تعطيل المشاركة العامة. سيسمح برنامج عام لأي مستخدم موثَّق بتشغيل كود مخزَّن عشوائي تحت رمزه الخاص — مشاركة برنامج محصورة بمؤسستك، وتعني دائمًا فقط "يمكن لزملاء الفريق مشاهدة نتيجته المخزَّنة مؤقتًا وتشغيل إعادة تنفيذ"، وليس أبدًا أن "الإنترنت العام يمكنه تنفيذه."
- **لا يمكن استدعاء actions التغيير والتنفيذ كأدوات من جسر الامتداد/iframe المعزول.** `save-data-program`، و`preview-data-program`، و`run-data-program`، و`delete-data-program` جميعها `toolCallable: false` — فقط الـ agent أو مسار قراءة من جانب الخادم يمكنه استدعاؤها، ولا يمكن أبدًا لـ iframe امتداد ضمن جلسة الفاتح استدعاؤها. `list-data-programs` و`get-data-program` للقراءة فقط ومحدَّدة النطاق بنفس مساعدات المشاركة المستخدَمة لأي مورد قابل للامتلاك (ownable) آخر.
- **البرامج محدَّدة النطاق بالتطبيق الذي أنشأها.** تمرير `appId` إلى عملية بحث يعني أن برنامجًا أُنشئ في تطبيق واحد غير قابل للتشغيل من دردشة agent تطبيق آخر في نشر قاعدة بيانات مشتركة، حتى لو كان المستدعي يملك خلاف ذلك وصولًا على مستوى الصف (مالك/مؤسسة/مشاركة) إليه.
- **اربط أو شاهد فقط برنامجًا تثق به.** بما أن التنفيذ يستخدم دائمًا بيانات اعتماد المُشاهِد نفسه، فإن مشاهدة نتيجة برنامج مشترك تشبه تمامًا فتح امتداد شخص آخر — الثقة يجب أن تكون في الكود نفسه، وليس فقط في منح الوصول.

## مثال Risk Meeting العملي {#risk-meeting-example}

يتضمن قالب analytics دراسة حالة عملية وقابلة للتشغيل: يقوم action الـ `ensure-risk-meeting-dashboard` بتثبيت برنامجَي بيانات بشكل متكرر الأثر (idempotent) بالإضافة إلى لوحة "Risk Meeting" ذات لوحتين، موضّحًا ثلاثة أمور لا يمكن لـ action مبرمَجة مسبقًا توقّعها لكل عميل.

**`risk-meeting-cohort.js`** — يبحث عن مجموعة قابلة للتهيئة من صفقات CRM بخاصية `risk_status` عشوائية (بحث IN على قيم مخصصة لا يمكن لـ action جاهزة توقّعه)، ويحل شركة كل صفقة عبر بحث ارتباط مُجزّأ (batched)، ويحل نطاق (domain) كل شركة، ويجلب معنويات حساب من مزوّد ثانٍ غير مرتبط، ويربط الاثنين حسب النطاق:

```js
// params: { riskStatuses?: string[], enterpriseOnly?: boolean }
const riskStatuses = params.riskStatuses?.length
  ? params.riskStatuses
  : ["On the Radar", "Churn Risk", "Confirmed Churn", "No Save Attempted"];

// 1. Arbitrary property IN-search — not expressible through a canned filter.
const dealSearch = await providerFetchAll("crm-provider", "/deals/search", {
  method: "POST",
  body: {
    filterGroups: [
      {
        filters: [
          { propertyName: "risk_status", operator: "IN", values: riskStatuses },
        ],
      },
    ],
  },
  itemsPath: "results",
  pagination: {
    cursorBodyPath: "after",
    nextCursorPath: "paging.next.after",
    maxPages: 20,
  },
});

// 2-3. Batched deal -> company -> domain resolution (elided here; see the
// real seed file for the full batching helper).

// 4. A second, unrelated provider — zero bespoke glue action required.
const accounts = await providerFetchAll("support-provider", "/accounts", {
  itemsPath: "data",
  pagination: {
    cursorParam: "cursor",
    nextCursorPath: "pagination.cursor",
    maxPages: 20,
  },
});

// 5. Join + emit.
const rows = dealSearch.items.map((deal) => ({
  deal_id: deal.id,
  risk_status: deal.properties.risk_status,
  arr: Number(deal.properties.total_contract_value || 0),
  // ...domain lookup + sentiment join...
}));

emit(rows, [
  { name: "deal_id", type: "string" },
  { name: "risk_status", type: "string" },
  { name: "arr", type: "number" },
]);
```

**`risk-meeting-pylon-early-warning.js`** — العرض المكمّل: الحسابات التي حُددت بالفعل كمعرَّضة للخطر بحسب معنويات الدعم والتي **لم تظهر بعد** في مجموعة CRM أعلاه. هذه الفجوة — إشارة دعم تسبق إشارة CRM — تتطلب استبعاد مجموعة مزوّد واحد من مجموعة مزوّد آخر، وهو بالضبط نوع منطق المجموعات عبر المزودين الذي لا يمكن لأي action خاصة بمزوّد واحد التعبير عنه.

يُثبَّت البرنامجان ويُحافَظ عليهما محدَّثين بواسطة `ensure-risk-meeting-dashboard`، والذي:

- يُحدّث كل برنامج حسب `name` الثابت الخاص به، بحيث تؤدي إعادة تشغيل المثبّت إلى تحديث نفس الصفوف بدلاً من تكرارها.
- لا يتطلب بيانات اعتماد CRM/الدعم وقت التثبيت — تُحل مصادقة المزوّد حسب المُشاهِد فقط عندما تُعرض لوحة فعليًا أو يُستدعى `run-data-program`، بما يتسق مع تحديد نطاق التخزين المؤقت حسب المُشاهِد أعلاه.
- يبني لوحة معلومات ذات لوحتين تستخدم لوحاتها `source: "program"`، مرتبطة بمعرّف كل برنامج.

راجع skill المسمّاة `data-programs` للحصول على الكود الكامل الموضَّح بالتعليقات.

## كيف تتبنى تطبيقات أخرى مصدر اللوحة {#adopting-the-panel-source}

مصدر لوحة dashboard من نوع `"program"` هو أحد أشكال تبنّي هذه البنية الأساسية، مُنفَّذ بالكامل في طبقة استعلام اللوحات الخاصة بالتطبيق المضيف نفسه — لا يوجد أي شيء خاص بلوحات المعلومات مدمج في `@agent-native/core/data-programs` نفسها. تطبيق قالب analytics هو مرجع مفيد لربط تطبيقك الخاص:

- يحمل عمود `sql` الخاص باللوحة واصف JSON صغير — `{ programId, params? }` — بدلاً من سلسلة استعلام.
- يستدعي عرض اللوحة `runDataProgram` مع `triggeredBy: "panel_view"` وهوية المُشاهِد الحالي، بحيث يُطبَّق التخزين المؤقت حسب المُشاهِد وحل بيانات الاعتماد تلقائيًا.
- عند النجاح، تحصل اللوحة على `{ rows, schema }` — بنفس الشكل الذي يعيده أي مصدر لوحة آخر، بحيث يتدفق دون تعديل عبر مكونات الرسم البياني/الجدول الحالية.
- عند الفشل، يُفضَّل `lastGoodRun` على لوحة معطوبة كلما وُجد؛ فقط عند عدم وجود نجاح سابق تعرض اللوحة خطأً منظّمًا (مع نص أكثر ودّية لرموز شائعة مثل `access_denied`، و`archived`، و`background_pending`).

يمكن لأي تطبيق تنفيذ نفس التبنّي للوحات المعلومات/الجداول الخاصة به: استدعِ `run-data-program` (أو `runDataProgram` مباشرة إذا كنت ضمن نفس عملية الخادم) من مسار القراءة الخاص بك، ومرِّر `rows`/`schema` المُعادة إلى مكونات العرض الخاصة بك.

## مستندات ذات صلة {#related}

- [الامتدادات](/docs/extensions) — البنية الأساسية للتطبيقات المصغّرة المعزولة التي تشارك برامج البيانات معها وضعية أمان ونموذج مشاركة، لكن بواجهة خاصة بها.
