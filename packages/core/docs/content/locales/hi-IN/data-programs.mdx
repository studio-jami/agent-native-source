# डेटा प्रोग्राम

एक **डेटा प्रोग्राम (data program)** एक नामित, सहेजा गया, एजेंट द्वारा लिखा गया JS स्क्रिप्ट है जो डेटा फ़ेच, जॉइन, फ़िल्टर या एग्रीगेट करता है — `providerFetch` / `providerFetchAll` / `providerSearchAll` के ज़रिए provider APIs, `appAction` के ज़रिए app डेटा, या Resources-बैक्ड workspace फ़ाइलें — और परिणाम के साथ बिल्कुल एक बार `emit(rows, schema)` कॉल करता है। परिणाम SQL में कैश होता है और कोई भी chart/table कंपोनेंट इसे रेंडर कर सकता है, जो हर पेज व्यू पर नहीं बल्कि आपकी चुनी हुई पॉलिसी पर रिफ्रेश होता है।

जहाँ भी [`run-code`](/docs/actions) उपलब्ध है वहाँ यह अपने आप उपलब्ध हो जाता है — प्रति app इसे माउंट या कॉन्फ़िगर करने की कोई ज़रूरत नहीं है। अगर आपके app में पहले से run-code सक्षम है, तो `save-data-program`, `preview-data-program`, `run-data-program`, `list-data-programs`, `get-data-program`, और `delete-data-program` पहले से ही रजिस्टर्ड actions हैं।

## यह क्यों मौजूद है {#why}

एक सहेजे गए डेटा प्रोग्राम के बिना, "X का लाइव व्यू बनाओ" अक्सर दो खराब जगहों में से किसी एक पर खत्म होता है:

- **प्रति vendor/क्वेरी-आकार एक हार्डकोडेड action।** नाज़ुक, और यह हर ग्राहक की कस्टम प्रॉपर्टीज़, provider-पार जॉइन, या किसी मनमाने फ़िल्टर का अनुमान नहीं लगा सकता जिसके बारे में आपने सोचा ही नहीं।
- **हर पेज व्यू पर वही फ़ेच/जॉइन लॉजिक शुरू से फिर चलाना।** धीमा, और हर बार जब कोई chart देखता है तो वही provider API कोटा फिर से खर्च होता है।

डेटा प्रोग्राम बीच का रास्ता है: फ़ेच/जॉइन/एग्रीगेशन लॉजिक को एक बार सामान्य `run-code` के रूप में लिखें, इसे सहेजें, और यह तय करने दें कैश और रिफ्रेश पॉलिसी को कि इसे वास्तव में कितनी बार फिर से चलाने की ज़रूरत है।

## लेखन वर्कफ़्लो {#authoring-workflow}

1. **पहले चैट में प्रोटोटाइप बनाएँ।** फ़ेच/जॉइन/एग्रीगेशन लॉजिक को एक सामान्य `run-code` स्क्रिप्ट की तरह लिखें और पुष्टि करें कि यह अपेक्षित पंक्तियाँ लौटाता है।
2. **इसे बदलें ताकि यह अंत में परिणाम लौटाने या प्रिंट करने के बजाय बिल्कुल एक बार `emit(rows, schema?)` कॉल करे।**
3. **इसे `save-data-program` से सहेजें।** यह सहेजने से पहले **`defaultParams` के साथ कोड का dry-run चलाता है** और अगर यह विफल होता है तो एक स्ट्रक्चर्ड एरर के साथ सेव को अस्वीकार कर देता है, ताकि कोई टूटा हुआ प्रोग्राम कभी सहेजा न जाए। सफल होने पर यह `{ programId, rowCount, columns, sampleRows }` लौटाता है — उपयोगकर्ता को यह बताने से पहले कि यह तैयार है, इसे इस बात के प्रमाण के रूप में लें कि प्रोग्राम अपेक्षित पंक्तियाँ बनाता है।
4. **किसी dashboard पैनल को इससे बाइंड करें**, या अपने खुद के रीड पाथ से सीधे `run-data-program` कॉल करें। इस प्रिमिटिव में खुद कोई dashboard-विशिष्ट वायरिंग बिल्ट-इन नहीं है — analytics टेम्पलेट का `"program"` पैनल सोर्स (नीचे) एक अपनाने का उदाहरण है, कोई अनिवार्य पैटर्न नहीं।
5. **बिना सहेजे इटरेट करें** `preview-data-program({ code, params? })` के ज़रिए — वही dry-run पाथ, कोई सहेजी गई पंक्ति नहीं।

### एक व्यावहारिक उदाहरण

मान लीजिए एक app किसी सपोर्ट provider से एक लाइव "हाल के उच्च-मूल्य वाले सपोर्ट टिकट्स" टेबल चाहता है, जो किसी ऐसे थ्रेशोल्ड से फ़िल्टर हो जिसे कोई बिल्ट-इन action उजागर नहीं करता:

```ts
// The program's stored `code` — ordinary run-code, with one emit() call.
const minValue = Number(params.minValue || 5000);

const tickets = await providerFetchAll("acme-support", "/v1/tickets", {
  query: { status: "open" },
  itemsPath: "data",
  pagination: {
    cursorParam: "cursor",
    nextCursorPath: "next_cursor",
    maxPages: 10,
  },
});

const rows = (tickets.items || [])
  .filter((t) => Number(t.contract_value || 0) >= minValue)
  .map((t) => ({
    ticket_id: t.id,
    subject: t.subject,
    account: t.account_name,
    contract_value: Number(t.contract_value || 0),
    opened_at: t.created_at,
  }));

emit(rows, [
  { name: "ticket_id", type: "string" },
  { name: "subject", type: "string" },
  { name: "account", type: "string" },
  { name: "contract_value", type: "number" },
  { name: "opened_at", type: "string" },
]);
```

इसे सहेजना:

```ts
await appAction("save-data-program", {
  name: "high-value-open-tickets",
  title: "High-Value Open Tickets",
  description:
    "Open support tickets above a configurable contract-value threshold.",
  code: "/* the script above */",
  defaultParams: { minValue: 5000 },
  refreshMode: "ttl",
  refreshTtlMs: 300_000,
});
```

प्रतिक्रिया में `{ programId, rowCount, columns, sampleRows }` होता है — dry-run के पूर्ण होने का प्रमाण। वहाँ से, `run-data-program({ programId, params: { minValue: 10000 } })` इसे एक अलग थ्रेशोल्ड के लिए फिर चलाता है (या कैश देता है), और एक dashboard पैनल उसी तरह `programId` से बाइंड हो सकता है।

## सैंडबॉक्स सतह {#sandbox-surface}

एक डेटा प्रोग्राम बिल्कुल उसी `executeSandboxCode` सैंडबॉक्स के ज़रिए चलता है जिससे `run-code` चलता है — इस प्रिमिटिव के लिए कोई नया सैंडबॉक्सिंग, क्रेडेंशियल, SSRF, या कोटा कोड मौजूद नहीं है। इसे वही ग्लोबल्स मिलते हैं (`providerFetch`, `providerFetchAll`, `providerSearchAll`, `appAction`, `webFetch`, `workspace*` Resources हेल्पर्स) साथ ही फ्रेमवर्क द्वारा इंजेक्ट किए गए एक छोटे प्रील्यूड से दो अतिरिक्त चीज़ें:

- **`params`** — इस रन के लिए params रखने वाला एक **फ़्रोज़न** ग्लोबल ऑब्जेक्ट (`defaultParams` को `run-data-program` या पैनल डिस्क्रिप्टर को पास की गई चीज़ के साथ मर्ज किया गया)। इसे पढ़ना ठीक है; इसे म्यूटेट करना एरर फेंकता है।
- **`emit(rows, schema?)`** — इसे अंत में **बिल्कुल एक बार** कॉल करें। दूसरी कॉल एरर फेंकती है। `rows` सामान्य ऑब्जेक्ट्स का एक array होना चाहिए। `schema`, जब दिया जाए, `{ name: string, type: string }[]` होता है; जब छोड़ दिया जाए तो यह पहली 50 पंक्तियों से अनुमानित होता है (यदि सैंपल में पंक्तियाँ primitive type पर असहमत हों तो कॉलम `"json"` होता है, अन्यथा `"number"` / `"string"` / `"boolean"`)। डिबगिंग के लिए `console.log` पूरी तरह मुक्त रहता है — रनर मिले-जुले stdout/stderr को कैप्चर करता है और परिणाम पार्स करते समय केवल वह सेंटिनल लाइन हटाता है जो `emit()` लिखता है।

हर रन पर लागू होने वाली सीमाएँ (प्रति-प्रोग्राम कॉन्फ़िगर करने योग्य नहीं):

| सीमा                                                  | मान                                                                                                                                                                         |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| अधिकतम emit की गई पंक्तियाँ                           | 10,000 — इससे आगे की पंक्तियाँ हटा दी जाती हैं, `truncated: true`                                                                                                           |
| अधिकतम emit किए गए परिणाम का आकार                     | 4 MiB — फ़िट करने के लिए अंत से पंक्तियाँ हटाई जाती हैं; सीमा से बड़ी एक भी पंक्ति एक कठोर `result_too_large` विफलता है, क्योंकि छोटा करने के लिए कुछ भी सुरक्षित नहीं बचता |
| प्रति app अधिकतम सक्रिय (गैर-आर्काइव्ड) प्रोग्राम     | 200 — जगह खाली करने के लिए अप्रयुक्त प्रोग्राम आर्काइव करें                                                                                                                 |
| न्यूनतम रिफ्रेश TTL                                   | 60,000 ms                                                                                                                                                                   |
| प्रति (प्रोग्राम, params, व्यूअर) रखी गई रन पंक्तियाँ | नवीनतम 5 — पुरानी पंक्तियाँ हर लेखन पर अपने आप हटा दी जाती हैं                                                                                                              |

ट्रंकेशन हमेशा स्पष्ट होता है (परिणाम में `truncated: true`) — कभी भी चुपचाप छोड़ा नहीं जाता।

## Params और dashboard पैनल बाइंडिंग {#params-and-panel-binding}

प्रोग्राम एक वैकल्पिक `paramsSchema` (स्वीकृत params बताने वाला JSON Schema) और `defaultParams` घोषित कर सकते हैं। एक dashboard पैनल के क्वेरी फ़ील्ड में क्वेरी स्ट्रिंग के बजाय एक छोटा JSON डिस्क्रिप्टर रखकर `"program"` पैनल सोर्स अपनाता है:

```json
{ "programId": "dp_a1b2c3d4e5f6", "params": { "minValue": 10000 } }
```

पैनल बिल्कुल उन्हीं chart/table कंपोनेंट्स के ज़रिए रेंडर होता है जिनसे कोई भी अन्य पैनल सोर्स होता है — इसे हमेशा केवल `{ rows, schema }` वापस मिलता है। किसी पैनल को किसी प्रोग्राम से बाइंड करना बस इतना ही है कि "कौन सा सहेजा गया प्रोग्राम, किन params ओवरराइड्स के साथ"; कहीं और कोई डुप्लिकेट dashboard-विशिष्ट execution पाथ मौजूद नहीं है।

## रिफ्रेश, TTL, और कैशिंग सिमेंटिक्स {#refresh-and-caching}

हर `(programId, paramsHash)` जोड़े का `data_program_runs` में अपना खुद का रन इतिहास होता है, और `paramsHash` केवल params ही नहीं बल्कि **व्यूअर** को भी शामिल करता है (कारण के लिए सुरक्षा सेक्शन देखें)। हर रीड पर क्या होता है:

| स्थिति                                                                                 | व्यवहार                                                                                                                                                                                          |
| -------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ताज़ा कैश (`refreshTtlMs` से नया, या `refreshMode: "manual"` के साथ पहले की कोई सफलता) | कैश की गई पंक्तियाँ तुरंत लौटाता है, `cacheHit: true`                                                                                                                                            |
| कैश TTL से पुराना है                                                                   | सिंक्रोनसली फिर से एक्ज़ीक्यूट होता है — पैनल-व्यू से ट्रिगर हुए रीड्स के लिए 25s का बजट, एजेंट/मैनुअल कॉल्स के लिए 120s — और सफलता पर कैश बदल देता है                                           |
| `refreshMode: "manual"`                                                                | कभी अपने आप रिफ्रेश नहीं होता; केवल स्पष्ट `forceRefresh: true` (या UI का मैनुअल-रिफ्रेश action) इसे फिर चलाता है                                                                                |
| अभी तक कोई कैश नहीं                                                                    | एक सामान्य पहली फ़ेच की तरह सिंक्रोनसली एक्ज़ीक्यूट होता है                                                                                                                                      |
| `background: true` प्रोग्राम, कोई ताज़ा कैश नहीं                                       | तुरंत `stale: true` के साथ आखिरी अच्छा रन देता है और एक टिकाऊ बैकग्राउंड एक्ज़ीक्यूशन को क्यू में डालता है (10-मिनट का बजट); एक बाद की कॉल एक्ज़ीक्यूशन पूरा होने पर परिणाम को अंतिम रूप देती है |
| बैकग्राउंड प्रोग्राम अभी भी चल रहा है, पहले कोई सफलता नहीं                             | `background_pending` विफलता — एक स्पष्ट "अभी भी गणना हो रही है" स्थिति दिखाएँ, कभी खाली परिणाम नहीं                                                                                              |
| एक्ज़ीक्यूशन विफल होता है (टाइमआउट, सैंडबॉक्स एरर, गलत `emit()` आकार)                  | एक स्ट्रक्चर्ड `{ code, message }` विफलता; अगर पहले की कोई सफलता मौजूद है, तो इसे `lastGoodRun` के रूप में जोड़ा जाता है ताकि पैनल खाली होने के बजाय पुराना डेटा दिखा सके                        |
| प्रोग्राम आर्काइव किया गया (`delete-data-program` के ज़रिए सॉफ्ट-डिलीट)                | स्पष्ट `archived` विफलता — कभी चुपचाप खाली पैनल नहीं                                                                                                                                             |

`background: true` का इस्तेमाल केवल उन प्रोग्रामों के लिए करें जो नियमित रूप से फ़ोरग्राउंड टाइमआउट से आगे निकल जाते हैं — बड़े मल्टी-provider जॉइन्स, गहरी पेजिनेशन। बाकी सब कुछ फ़ोरग्राउंड में ही रहना चाहिए: यह समझने में आसान है और तेज़ी से ताज़ा डेटा देता है।

### कैश व्यूअर के अनुसार सीमित क्यों है {#per-viewer-cache}

किसी प्रोग्राम के अंदर `providerFetch` **कॉल करने वाले व्यूअर के अपने रिक्वेस्ट कॉन्टेक्स्ट** का उपयोग करके ऑथ हल करता है — कभी प्रोग्राम के लेखक का नहीं। यह सच है चाहे कॉल एजेंट से आए, `run-data-program` कॉल से आए, या किसी टीममेट द्वारा देखे जा रहे dashboard पैनल से आए। एक ही प्रोग्राम को एक ही params के साथ चलाने वाले दो अलग-अलग व्यूअर वैध रूप से दो अलग परिणाम पा सकते हैं: एक के पास कॉन्फ़िगर्ड HubSpot key है और दूसरे के पास नहीं, या row-स्तर की provider अनुमतियाँ व्यक्ति के अनुसार अलग हैं।

इसी वजह से, `hashDataProgramParams` कैनोनिकलाइज़्ड params के साथ-साथ कॉलर की पहचान को भी कैश-की में शामिल करता है। यह **शेयर्ड dashboards** के लिए सबसे ज़्यादा मायने रखता है: अगर कैश केवल params से की-ड होता, तो पैनल देखने वाला पहला व्यक्ति चुपचाप तय कर देता कि हर अन्य व्यूअर क्या देखेगा, उन व्यूअर्स समेत जिनके पास वे क्रेडेंशियल्स नहीं हैं जिन्होंने वह कैश किया गया परिणाम बनाया। प्रति-व्यूअर स्कोपिंग का मतलब है कि क्रेडेंशियल न रखने वाला टीममेट उस पैनल पर किसी और का डेटा नहीं बल्कि अपनी खुद की ऑथ एरर देखता है।

## सुरक्षा मॉडल {#security}

<Callout id="doc-block-dp-security" tone="success">

**रॉ provider टोकन कभी भी मॉडल या ब्राउज़र तक नहीं पहुँचते।** सैंडबॉक्स से बाहर निकलने वाली एकमात्र चीज़ `{ rows, schema }` है जो एक `emit()` कॉल बनाता है — बाकी हर run-code एक्ज़ीक्यूशन और हर अन्य dashboard पैनल सोर्स जैसी ही ट्रस्ट बाउंड्री।

</Callout>

- **केवल सर्वर-साइड।** एक डेटा प्रोग्राम का कोड विशेष रूप से सर्वर पर `executeSandboxCode` के अंदर ही चलता है। न तो सहेजा गया सोर्स और न ही इसके द्वारा छुई गई कोई क्रेडेंशियल ब्राउज़र या मॉडल को भेजी जाती है — केवल emit की गई पंक्तियाँ/schema और संक्षिप्त रन मेटाडेटा (row count, columns, sample rows) ही भेजे जाते हैं।
- **क्रेडेंशियल्स हमेशा कॉल करने वाले व्यूअर की होती हैं, कभी प्रोग्राम के मूल लेखक की नहीं।** यह कैश-की को भी क्यों तय करता है, इसके लिए ऊपर [रिफ्रेश, TTL, और कैशिंग सिमेंटिक्स](#per-viewer-cache) देखें।
- **`allowPublic: false` एक कठोर invariant है, कोई डिफ़ॉल्ट नहीं।** डेटा प्रोग्राम शेयरिंग रजिस्ट्री में उसी तरह रजिस्टर होते हैं जैसे [एक्सटेंशन](/docs/extensions#security), पब्लिक शेयरिंग बंद रखते हुए। एक पब्लिक प्रोग्राम किसी भी ऑथेंटिकेटेड उपयोगकर्ता को अपने खुद के टोकन के तहत मनमाना सहेजा गया कोड चलाने देगा — किसी प्रोग्राम को शेयर करना आपके org तक सीमित है, और इसका मतलब हमेशा सिर्फ यही है कि "टीममेट्स इसके कैश किए गए परिणाम को देख सकते हैं और फिर से रन ट्रिगर कर सकते हैं," कभी यह नहीं कि "सामान्य इंटरनेट इसे चला सकता है।"
- **म्यूटेट करने और एक्ज़ीक्यूट करने वाली actions सैंडबॉक्स्ड एक्सटेंशन/iframe ब्रिज से टूल के रूप में कॉल करने योग्य नहीं हैं।** `save-data-program`, `preview-data-program`, `run-data-program`, और `delete-data-program` सभी `toolCallable: false` हैं — केवल एजेंट या एक सर्वर-साइड रीड पाथ ही इन्हें कॉल कर सकता है, कभी ओपनर के सेशन के तहत कोई एक्सटेंशन iframe नहीं। `list-data-programs` और `get-data-program` रीड-ओनली हैं और किसी भी अन्य ownable रिसोर्स की तरह उन्हीं शेयरिंग हेल्पर्स के ज़रिए एक्सेस-स्कोप्ड हैं।
- **प्रोग्राम उस app तक सीमित हैं जिसने उन्हें बनाया।** किसी लुकअप को `appId` पास करने का मतलब है कि एक app में बनाया गया प्रोग्राम शेयर्ड-डेटाबेस डिप्लॉयमेंट में किसी दूसरे app के एजेंट चैट से रन करने योग्य नहीं है, भले ही कॉलर के पास अन्यथा row-स्तर (owner/org/share) एक्सेस मौजूद हो।
- **केवल उसी प्रोग्राम से बाइंड करें या देखें जिस पर आप भरोसा करते हैं।** क्योंकि एक्ज़ीक्यूशन हमेशा व्यूअर की अपनी क्रेडेंशियल्स का उपयोग करता है, किसी शेयर्ड प्रोग्राम का परिणाम देखना बिल्कुल किसी और का एक्सटेंशन खोलने जैसा है — भरोसा कोड पर करें, केवल एक्सेस ग्रांट पर नहीं।

## Risk Meeting कार्यशील उदाहरण {#risk-meeting-example}

analytics टेम्पलेट में एक ठोस, चलाने योग्य केस स्टडी शामिल है: `ensure-risk-meeting-dashboard` action दो डेटा प्रोग्राम और एक दो-पैनल "Risk Meeting" dashboard को इडेम्पोटेंटली इंस्टॉल करता है, यह दिखाते हुए कि तीन चीज़ें एक हार्डकोडेड action हर ग्राहक के लिए अनुमानित नहीं कर सकता।

**`risk-meeting-cohort.js`** — एक मनमाने `risk_status` प्रॉपर्टी (एक कस्टम वैल्यूज़ पर IN-सर्च जिसे कोई तैयार action अनुमानित नहीं कर सकता) से CRM deals का एक कॉन्फ़िगर करने योग्य कोहॉर्ट खोजता है, बैच्ड एसोसिएशन लुकअप के ज़रिए हर deal की कंपनी हल करता है, हर कंपनी का डोमेन हल करता है, दूसरे, असंबंधित provider का account sentiment फ़ेच करता है, और डोमेन से दोनों को जॉइन करता है:

```js
// params: { riskStatuses?: string[], enterpriseOnly?: boolean }
const riskStatuses = params.riskStatuses?.length
  ? params.riskStatuses
  : ["On the Radar", "Churn Risk", "Confirmed Churn", "No Save Attempted"];

// 1. Arbitrary property IN-search — not expressible through a canned filter.
const dealSearch = await providerFetchAll("crm-provider", "/deals/search", {
  method: "POST",
  body: {
    filterGroups: [
      {
        filters: [
          { propertyName: "risk_status", operator: "IN", values: riskStatuses },
        ],
      },
    ],
  },
  itemsPath: "results",
  pagination: {
    cursorBodyPath: "after",
    nextCursorPath: "paging.next.after",
    maxPages: 20,
  },
});

// 2-3. Batched deal -> company -> domain resolution (elided here; see the
// real seed file for the full batching helper).

// 4. A second, unrelated provider — zero bespoke glue action required.
const accounts = await providerFetchAll("support-provider", "/accounts", {
  itemsPath: "data",
  pagination: {
    cursorParam: "cursor",
    nextCursorPath: "pagination.cursor",
    maxPages: 20,
  },
});

// 5. Join + emit.
const rows = dealSearch.items.map((deal) => ({
  deal_id: deal.id,
  risk_status: deal.properties.risk_status,
  arr: Number(deal.properties.total_contract_value || 0),
  // ...domain lookup + sentiment join...
}));

emit(rows, [
  { name: "deal_id", type: "string" },
  { name: "risk_status", type: "string" },
  { name: "arr", type: "number" },
]);
```

**`risk-meeting-pylon-early-warning.js`** — पूरक व्यू: वे accounts जो सपोर्ट sentiment द्वारा पहले से ही जोखिम भरे चिह्नित हैं लेकिन ऊपर वाले CRM कोहॉर्ट में **अभी तक** नहीं दिखे। यह अंतर — CRM सिग्नल से आगे निकलता सपोर्ट सिग्नल — एक provider के कोहॉर्ट को दूसरे से बाहर रखने की ज़रूरत रखता है, जो ठीक उसी तरह का provider-पार सेट लॉजिक है जिसे कोई सिंगल-vendor action व्यक्त नहीं कर सकता।

दोनों प्रोग्राम `ensure-risk-meeting-dashboard` द्वारा इंस्टॉल किए जाते हैं और अपडेट रखे जाते हैं, जो:

- हर प्रोग्राम को उसके स्थिर `name` से अपडेट करता है, ताकि इंस्टॉलर को फिर से चलाने पर वही पंक्तियाँ डुप्लिकेट होने के बजाय अपडेट हों।
- इंस्टॉल के समय किसी CRM/सपोर्ट क्रेडेंशियल की ज़रूरत नहीं होती — provider ऑथ प्रति-व्यूअर तभी हल होता है जब कोई पैनल वास्तव में रेंडर होता है या `run-data-program` कॉल किया जाता है, जो ऊपर बताए गए प्रति-व्यूअर कैश स्कोपिंग के अनुरूप है।
- एक दो-पैनल dashboard बनाता है जिसके पैनल `source: "program"` इस्तेमाल करते हैं, जो हर प्रोग्राम के id से बाइंड होते हैं।

पूरी तरह से एनोटेटेड सोर्स के लिए `data-programs` skill देखें।

## अन्य apps पैनल सोर्स को कैसे अपनाते हैं {#adopting-the-panel-source}

`"program"` dashboard पैनल सोर्स इस प्रिमिटिव को अपनाने का एक उदाहरण है, जो पूरी तरह से होस्ट app की अपनी पैनल-क्वेरी लेयर में लागू किया गया है — `@agent-native/core/data-programs` में खुद कुछ भी dashboard-विशिष्ट बिल्ट-इन नहीं है। analytics टेम्पलेट का इम्प्लीमेंटेशन आपके अपने को वायर करने के लिए एक उपयोगी संदर्भ है:

- पैनल का `sql` कॉलम क्वेरी स्ट्रिंग के बजाय एक छोटा JSON डिस्क्रिप्टर — `{ programId, params? }` — रखता है।
- किसी पैनल को रेंडर करना `triggeredBy: "panel_view"` और वर्तमान व्यूअर की पहचान के साथ `runDataProgram` कॉल करता है, ताकि प्रति-व्यूअर कैश और क्रेडेंशियल रिज़ॉल्यूशन अपने आप लागू हो जाए।
- सफलता पर, पैनल को `{ rows, schema }` मिलता है — बिल्कुल वही आकार जो हर अन्य पैनल सोर्स लौटाता है, ताकि यह मौजूदा chart/table कंपोनेंट्स से बिना किसी बदलाव के प्रवाहित हो सके।
- विफलता पर, जब भी कोई `lastGoodRun` मौजूद हो तो उसे टूटे हुए पैनल पर प्राथमिकता दी जाती है; केवल तब जब पहले की कोई सफलता न हो, पैनल एक स्ट्रक्चर्ड एरर दिखाता है (आम कोड्स जैसे `access_denied`, `archived`, और `background_pending` के लिए ज़्यादा दोस्ताना टेक्स्ट के साथ)।

कोई भी app अपने खुद के dashboards/tables के लिए यही तरीका अपना सकता है: अपने खुद के रीड पाथ से `run-data-program` (या अगर वही सर्वर प्रोसेस के अंदर हैं तो सीधे `runDataProgram`) कॉल करें, और लौटाए गए `rows`/`schema` को अपने खुद के रेंडरिंग कंपोनेंट्स को सौंप दें।
